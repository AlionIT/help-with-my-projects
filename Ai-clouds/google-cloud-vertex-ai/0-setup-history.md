# ❌ История попыток и антипаттерны (Как делать НЕ НАДО)

Этот документ жизненно важен для понимания того, почему мы используем сложную схему авторизации (ADC), а не простой текстовый ключ-пароль.

## Тупик: Попытка использовать классический API Key
Изначально была предпринята интуитивная попытка создать простой API-ключ для доступа к моделям. 

**Что мы делали:**
1. Зашли в Google Cloud Console -> **APIs & Services** -> **Credentials**.
2. Создали **API Key**.
3. Перешли в раздел *API restrictions* (Ограничения API), чтобы в целях безопасности привязать ключ только к нейросетям.

**В чем заключалась проблема:**
В гигантском списке доступных сервисов нужный нам пункт **Vertex AI API** оказался неактивным (серого цвета). Выбрать его было физически невозможно. 
При наведении на неактивный чекбокс система выдавала подсказку:
> *"This API requires authentication with a service account bound API key"*

**Суть проблемы (почему так):**
Платформа Vertex AI предназначена для корпоративного сектора (Enterprise). Google Cloud считает эти модели слишком мощными и приватными, чтобы выдавать к ним доступ через простой текстовый токен, который можно случайно "запушить" в публичный репозиторий GitHub. Интерфейс Google Cloud работает по принципу "Secure by default".

**Итог и правило для агентов:**
**НИКОГДА** не пытайтесь сгенерировать или использовать обычный API-ключ для работы с Vertex AI. Единственный рабочий путь — это механизм **ADC (Application Default Credentials)** и **Service Accounts** (описано в `1-overview.md`).
